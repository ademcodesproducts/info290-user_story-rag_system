TITLE: Query Performance Explainability for Databricks SQL

PERSONA:
Role: Analytics Engineer / Data Analyst
Company size: 100–10,000 employees
Context: Runs queries against Delta tables via SQL warehouses. Experiences significant and unpredictable query latency variance — the same query can take 5 seconds one run and 3 minutes the next. Has no visibility into why, and no self-service path to diagnose or improve performance.
Job-to-be-done: Understand why a specific query was slow — which stage, which table scan, which join — without needing to read Spark query plans or open a support ticket.

PAIN POINTS:
1. "I ran the same query twice. Five seconds, then three minutes. Same data, same warehouse. Nobody can tell me why." — Analytics Engineer, financial services (INT-010, RETRO-003).
2. The Databricks SQL query history shows execution time but not a breakdown by stage — users see total duration but not whether the time was spent on compilation, I/O, shuffle, or join.
3. EXPLAIN output in Databricks SQL is a raw Spark physical plan — useful for experts but incomprehensible to the analytics engineers and analysts who need it most.
4. There is no in-product guidance connecting a slow query to an actionable fix — users who see a slow query have no path to "here is why this was slow and here is what to change."

REQUIREMENTS:
1. Query timeline breakdown: every completed query in the SQL query history shows a visual timeline bar divided into phases: compilation (ms), warehouse startup (ms, if applicable), planning (ms), I/O (ms), compute (ms), result transfer (ms). Clicking a phase shows its duration and percentage of total time.
2. Plain-language performance summary: after each query, a one-paragraph natural language summary of the dominant performance factor. Examples: "This query scanned 847GB because the filter on order_date did not use the Delta table's Z-ORDER index. Adding OPTIMIZE ... ZORDER BY (order_date) could reduce scan size by ~80%." Or: "This query was slow because the warehouse was starting from cold. The warehouse was idle for 45 minutes before this query ran."
3. Actionable recommendations: the performance summary includes 1–3 ranked, specific recommendations with estimated impact. Each recommendation links to relevant documentation.
4. Historical variance view: the query detail page shows a sparkline of the last 30 executions of this query (identified by query hash), with latency on the y-axis. Outlier runs are highlighted and their performance summary is accessible with one click.
5. Table health indicators: the query detail page flags if any scanned table has known performance issues: missing OPTIMIZE (last optimized >7 days ago), missing Z-ORDER on a frequently-filtered column (detected from query history patterns), or high file count (>10,000 small files).

SUCCESS CRITERIA:
- An analytics engineer can identify the primary cause of a slow query in under 3 minutes using the query timeline and plain-language summary, without reading the Spark physical plan.
- Plain-language performance summaries correctly identify the dominant performance factor (compilation, I/O, warehouse cold start, shuffle, etc.) for >80% of queries in a benchmark set of 100 test queries with known root causes.
- Table health indicator for "missing OPTIMIZE" fires for >95% of tables that have not been optimized in 7+ days.

OPEN QUESTIONS:
- Should the plain-language summary be generated by an LLM (Mosaic AI) or by a rule-based system? LLM gives better quality but adds latency and cost per query.
- How do we handle queries where the performance bottleneck is outside Databricks (e.g., slow JDBC source or upstream API call)?
