TITLE: Feature Store Temporal Validation and Data Leakage Detection

PERSONA:
Role: ML Engineer / Data Scientist
Company size: 300–10,000 employees
Context: Training supervised ML models using point-in-time feature lookups from the Databricks Feature Store. Needs confidence that training data does not contain future-state feature values relative to the label timestamp (look-ahead bias).
Job-to-be-done: Know immediately — at training time, not 6 weeks into production — if a feature join has introduced temporal data leakage.

PAIN POINTS:
1. "We found a data leakage issue 6 weeks after deployment because our precision dropped from 0.91 to 0.84. Six weeks of a wrong fraud detection model. The Feature Store joined on the wrong timestamp and said nothing." — ML Engineer, insurance company (TKT-007, INT-006).
2. The FeatureLookup API accepts a timestamp_lookup_key parameter but does not validate whether the joined feature values predate the event timestamp — it silently joins regardless.
3. Data leakage from look-ahead bias is the #1 silent model quality failure pattern in supervised learning, yet the Feature Store has no detection layer.
4. Post-hoc leakage detection requires custom code (comparing feature_ts distributions against event_ts), which most teams don't write until after a production incident.

REQUIREMENTS:
1. Temporal validation parameter on FeatureLookup: leakage_check="warn" | "error" | "off" (default: "warn"). When enabled, after the training set join, Databricks computes the fraction of rows where any feature_timestamp > event_timestamp.
2. Warning threshold: if >1% of rows have future-state features, emit a DataLeakageWarning with the table name, column name, and percentage of affected rows.
3. Error threshold: if >10% of rows have future-state features and leakage_check="error", raise DataLeakageError and halt training set creation.
4. Leakage report artifact: log a JSON artifact to the associated MLflow run containing per-feature temporal distribution statistics (mean lag, p5, p95 of feature_ts - event_ts).
5. Documentation update: add a "Common pitfalls: temporal leakage" section to the Feature Store training data creation guide with worked examples of correct and incorrect timestamp_lookup_key specification.

SUCCESS CRITERIA:
- A training set with >1% future-state features always produces a visible warning in the notebook output and in the MLflow run UI — no silent incorrect behavior.
- The leakage check adds no more than 60 seconds of overhead for training sets up to 10M rows.
- Zero new data leakage incidents attributable to Feature Store point-in-time joins in the 6 months post-launch (tracked via support ticket analysis).

OPEN QUESTIONS:
- Should the leakage check run on a sample (e.g., 1% of rows) for performance, or always on the full dataset?
- How do we handle features that intentionally lag (e.g., 7-day rolling averages) where feature_ts will always predate event_ts by design?
