[PRD] MLflow Experiment Comparison View with Model Lineage Tracking

TL;DR
Data Scientists struggle to compare hundreds of MLflow runs across different notebooks or versions of code, often losing track of which data version produced which metric. This feature introduces a high-density, side-by-side comparison dashboard that links MLflow runs directly to their Unity Catalog data lineage and Git commit hash.

---
Problem Definition, Opportunity, and Outcome

The current MLflow UI is excellent for tracking individual runs but fails when a Data Scientist needs to compare metrics across different "Experiments" or when they need to see if a model's performance was due to a code change or a data shift in Delta Lake.

Users currently "hack" this by exporting run data to Excel or using multiple browser tabs. This is costly because it hides the "Data Lineage"—if a model was trained on a version of a table that has since been `VACUUM`ed or updated, it's impossible to reproduce. By integrating MLflow runs with Unity Catalog lineage and Repos, we ensure "Perfect Reproducibility."

Users & Use Cases

1. PRIMARY USERS: Data Scientists, ML Engineers
   • A Data Scientist is tuning a BERT model and has 50 runs with varying hyperparameters.
   • [Pain]: Can't easily see a "Difference" view that highlights only the changed parameters.
   • An ML Engineer needs to audit a model in production.
   • [Pain]: Can't see exactly which version of the `training_set` in Unity Catalog was used to generate the model artifact.

2. SECONDARY USERS (Edge Cases): Compliance/Audit
   • An auditor needs to verify that a model wasn't trained on "PII" data.
   • [Pain]: No easy way to trace from the Model Registry back to the source table's sensitivity tags.

Opportunity
This positions Databricks as the only "End-to-End" ML platform that treats data versioning (Delta) and model versioning (MLflow) as a single unit. It goes beyond Weights & Biases or Neptune by leveraging the native Unity Catalog integration.

JTDBs
• When comparing runs, I want to see a heatmap of my metrics, so I can instantly identify the best hyperparameters.
• When looking at a model in the Registry, I want to see the exact Delta Table Version (timestamp) used for training.
• When I have a "Best Run," I want to see the specific Git commit from Databricks Repos that generated it.
• When I have many failed runs, I want to hide them from my comparison view based on a metric threshold (e.g., "Accuracy > 0.5").

Requirements and Product Definition

M0: Advanced Comparison & Git Linking - Min Viable

P0: Parallel Coordinates Plot & Heatmap
   • Integrated UI within MLflow to visualize hyperparameter sweeps.
   • Highlighting of "Winning" runs based on user-selected metrics.

P0: Git Commit Attribution
   • Automatically capture and display the Databricks Repo commit hash in the MLflow run sidebar.

M1: Unity Catalog Integration - Advanced Usability

P0: Data Lineage Link
   • In the MLflow "Artifacts" tab, display the source Delta Table name and version number (`versionAsOf`).
   • Clickable link to the Catalog Explorer for that specific table version.

P1: "Diff" View for Parameters
   • A toggle that hides all parameters that are identical across selected runs, showing only the deltas.

P2: Multi-Experiment Comparison
   • Ability to select runs from different MLflow Experiments (e.g., "Project_A_Phase_1" and "Project_A_Phase_2") and compare them in one view.

User Experience: Bella, an ML Engineer, is comparing three versions of a churn model.
   • (J1) She selects the three best runs in the MLflow UI and clicks "Deep Compare."
   • (J2) The "Diff" view shows that the only difference was the `learning_rate` and the `source_table` version.
   • (J3) She clicks the source table link and sees that the second run used a version of the data that included a new "Region" column.
   • (J4) Satisfied, she promotes the run to "Production" in the Model Registry, knowing the full lineage is captured.
