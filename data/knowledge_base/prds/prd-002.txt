TITLE: Mosaic AI Playground — Prompt Versioning and Experiment Tracking

PERSONA:
Role: ML Engineer / Prompt Engineer
Company size: 200–2,000 employees
Context: Iterating on system prompts and model configurations for a production LLM application. Uses Mosaic AI Playground for interactive testing but has no way to track which prompt version produced which output quality.
Job-to-be-done: Version-control prompts, compare outputs across model versions and prompt variants, and promote a validated prompt to production with an audit trail.

PAIN POINTS:
1. "I've rewritten the same system prompt 40 times and I have no idea which version is live in production right now. It's in a Confluence page somewhere." — ML Engineer, Series B startup, user interview.
2. Playground sessions are ephemeral — closing the browser tab loses all prompt iterations with no save or history.
3. No way to run the same prompt against multiple models (e.g., DBRX vs. Llama 3 vs. external OpenAI endpoint) side-by-side in a single view.
4. Output quality evaluation is manual — engineers copy-paste outputs into a spreadsheet and rate them subjectively with no statistical tracking.
5. No connection between Playground iterations and MLflow — prompt experiments don't appear alongside model experiments in the same tracking UI.

REQUIREMENTS:
1. Prompt version history: every Playground session auto-saves prompt text, model config (model ID, temperature, max_tokens, system prompt), and timestamp. Users can name and tag versions (e.g., "v3-concise-tone", "prod-candidate").
2. Side-by-side comparison: run one prompt against up to 4 model configurations simultaneously; outputs shown in a grid with latency and token cost displayed per cell.
3. Evaluation rubrics: users define a rubric (e.g., "Correctness 1–5, Tone 1–5") and rate outputs inline; ratings stored and aggregable across runs.
4. MLflow integration: each saved Playground session logs to an MLflow experiment as a run, with prompt text as a parameter, model ID as a tag, and rubric scores as metrics.
5. Promote to production: a "Promote" button on a named prompt version creates a registered artifact in the Unity Catalog with the prompt text, model config, and evaluation history attached.

SUCCESS CRITERIA:
- An ML engineer can retrieve the exact prompt text and model config used in any production LLM call from the previous 90 days in under 2 minutes.
- Side-by-side comparison of 4 model configs completes and renders in under 10 seconds for prompts under 2,000 tokens.
- Promoted prompt artifacts appear in Unity Catalog lineage alongside the model they target.

OPEN QUESTIONS:
- Should prompt versions be stored in Unity Catalog Volumes or as MLflow artifacts?
- How do we handle prompts containing customer PII in the version history?
