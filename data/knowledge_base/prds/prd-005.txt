TITLE: SQL Warehouse Budget Controls and Cost Guardrails

PERSONA:
Role: Analytics Engineer / Platform Admin
Company size: 200–5,000 employees
Context: Manages SQL warehouse allocation for a team of 20–100 analysts. Serverless warehouses provide great auto-scaling but unpredictable costs — a single poorly-written query can trigger a warehouse scale-up that costs hundreds of dollars.
Job-to-be-done: Set per-warehouse and per-user spending guardrails so that cost surprises are eliminated without throttling analyst productivity.

PAIN POINTS:
1. "With serverless warehouses it scales up automatically, which is great, but I've had jobs that ran unexpectedly expensive because they triggered a warehouse scale-up I didn't anticipate." — Analytics Engineer, mid-size SaaS (INT-009).
2. No per-warehouse monthly budget cap exists — a warehouse can accumulate unlimited cost within a billing period.
3. Cost alerts for SQL warehouses are not available at the warehouse level — only at the workspace billing aggregate level, which is too coarse.
4. Analysts have no visibility into the cost of their individual queries before or after execution, making it impossible for them to self-govern.
5. Warehouse auto-stop settings (idle timeout) are the only cost control available, and they don't help when a warehouse is actively running expensive queries.

REQUIREMENTS:
1. Per-warehouse budget cap: admins set a monthly DBU budget per warehouse. At 80% of budget, a warning banner appears in the Databricks SQL UI for all users of that warehouse. At 100%, new queries are queued (not silently accepted) with a user-facing message: "Warehouse [name] has reached its monthly budget. Queries will resume on [date] or when budget is increased by your admin."
2. Per-user query cost estimate: before query execution, display an estimated DBU cost based on query plan analysis (similar to BigQuery's dry-run cost estimate). Users can choose to proceed or cancel.
3. Post-execution cost receipt: after a query completes, show the actual DBU cost in the query result panel. Accessible in query history.
4. Cost alerts: admins configure email/webhook alerts at configurable budget thresholds (50%, 80%, 100%) per warehouse. Alert includes: warehouse name, current spend, budget, top 3 queries by cost in current period.
5. Query cost leaderboard (opt-in): admins can enable a team-visible leaderboard showing top query costs for the current month — for healthy self-governance.

SUCCESS CRITERIA:
- Zero unplanned warehouse cost overruns in the 30 days post-launch for teams with budget caps enabled.
- Analysts can see the estimated cost of any query before running it in under 2 seconds (pre-execution estimate displayed without running the full query).
- Admins receive budget threshold alerts within 15 minutes of threshold crossing.

OPEN QUESTIONS:
- Should query cost estimates be shown by default (opt-out) or hidden by default (opt-in)?
- How do we handle queries that exceed a single-query cost threshold (e.g., a query that would cost $500)?
