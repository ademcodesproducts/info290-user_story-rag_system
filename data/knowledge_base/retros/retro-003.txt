PAIN POINTS RAISED:

1. Farida: "We finished the dbt model migration this sprint and the catalog looks great — beautiful table structure, clean schema. And zero column descriptions. Everything we wrote in schema.yml just... isn't there. Our analysts opened the catalog on day one and said 'all the descriptions are missing.' I had to explain that this is a known adapter gap, which is not a fun conversation."

2. Jordan: "Three analysts this sprint reported the same thing: they ran a query, it took 5 seconds. They ran it again 20 minutes later, 3 minutes. Same query, same table, same warehouse. When I investigated it turned out the warehouse had scaled down and the query recompilation wasn't cached. But we have no way to tell users this is happening or why. They just think the platform is unreliable."

3. Binta: "Two of us were working on the revenue attribution query simultaneously. We didn't know — no indicator in the UI that it was open. I saved my version and overwrote Kwame's 45 minutes of work. We only found out when Kwame tried to run his version and got my changes. There's no version history to recover from."

DECISIONS:
- Farida to implement the ALTER TABLE post-hook workaround for column comment sync. Accepted as technical debt — log it in backlog.
- Team will adopt a naming convention for shared queries in the SQL workspace (prefix: SHARED_) and add a note in the team wiki asking people to Slack before editing shared queries.

OPEN QUESTION:
Is there a Databricks SQL setting for query result caching TTL that we can tune to reduce the variance analysts are experiencing? Kwame to investigate.
