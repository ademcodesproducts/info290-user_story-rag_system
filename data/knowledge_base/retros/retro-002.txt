PAIN POINTS RAISED:

1. Andile: "I'm still manually building experiment comparison dashboards because the MLflow UI can't handle comparing 50 runs across 10 hyperparameters legibly. This sprint I spent a full day rebuilding the same dashboard I built last quarter because someone deleted the queries. This should be a native feature."

2. Mei-Ling: "We had two GPU training jobs fail this sprint because of spot preemption. Both were 8-hour runs. Both restarted from scratch because the Databricks job system doesn't know about our PyTorch checkpoints. We need checkpoint-aware retry — restart from the last checkpoint, not from epoch 0."

3. Carlos: "The batch inference UDF performance is still inconsistent. Same model, same cluster config, same data size — runtime varied from 18 minutes to 2 hours across three runs this sprint. I opened a support ticket two weeks ago. The response was 'it depends on partition layout.' Not actionable."

DECISIONS:
- Mei-Ling to add a post-run hook to all GPU training jobs that writes the latest checkpoint path to a job output parameter — so manual reruns can be seeded from the checkpoint even if Databricks doesn't do it automatically.
- Team agreed to standardize on mlflow.set_experiment() with a consistent naming convention (team/project/model_name) before next sprint — current experiment naming is chaos.

OPEN QUESTION:
Should we evaluate moving batch inference to a dedicated Databricks Jobs Compute cluster with consistent partition configuration instead of using shared interactive clusters? Yuna to cost-model.
