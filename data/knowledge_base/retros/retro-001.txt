PAIN POINTS RAISED:

1. Marcus: "We had three DLT pipeline failures this sprint and every single one gave us the same useless error message. 'Unexpected runtime error.' Three different root causes — schema drift, a bad S3 file, a metastore blip — but we couldn't tell from the error which was which. I burned 4 hours on what turned out to be a single malformed Parquet file."

2. Priya: "The retry logic on our Workflows DAG still isn't right. We set max_retries: 3 but when a task fails due to a transient network issue, the job retries from the beginning of the whole DAG, not from the failed task. We've got tasks that take 40 minutes upstream of the failure. Three retries means we waste 2 hours re-running work that already succeeded."

3. Devon: "I tried to implement conditional branching this sprint — 'if the row count check task returns zero, skip the downstream transforms.' The only way I could do it was notebook exit values parsed by downstream tasks as strings. It works but it's a hack. Someone's going to break it the first time they rename a variable."

DECISIONS:
- Team will add a wrapper function around all DLT pipeline triggers that logs the full event_log() output to a Delta table on failure — workaround until Databricks improves error messages.
- Priya to open a feature request ticket for task-level retry (not job-level) in Databricks Workflows.

OPEN QUESTION:
Is there a supported pattern for conditional branching in Workflows that doesn't use notebook exit value hacks? Fatima to check docs and Databricks Slack community before next sprint.
